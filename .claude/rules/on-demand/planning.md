# 実行計画書のルール

## ディレクトリ構成

| ディレクトリ    | 用途                           |
| --------------- | ------------------------------ |
| `VIBES/1_plan/` | 未着手の計画書                 |
| `VIBES/2_wip/`  | 作業中の計画書（進捗情報付き） |
| `VIBES/3_done/` | 完了した計画書（アーカイブ）   |

**基本原則:**

- 1タスク = 1ファイル
- ファイルの場所（plan/wip/done）がステータスを表す
- 計画と進捗は同一ファイル内で管理

## 命名規則

- `YYYYMMDD-{その日付の何個目か}-{日本語で簡単な作業内容}.md`
- 同じファイル名のまま plan → wip → done と移動する

**例:**

- `20260208-1-Redisリファクタ.md`
- `20260208-2-フィードバック機能修正.md`

## ライフサイクル

### 1. 計画作成（未着手）

`VIBES/1_plan/` にファイルを作成

```markdown
# タスク名

## 目的・背景

## 実装計画

## 技術選定
```

### 2. 作業開始（着手）

`1_plan/` → `2_wip/` に移動し、末尾に「## 進捗」セクションを追加

```bash
git mv VIBES/1_plan/ファイル名.md VIBES/2_wip/ファイル名.md
```

```markdown
# タスク名

## 目的

## 背景

### issue #{num}の要約

## 実装計画

## 技術選定

## 参考資料

公式docs、有用な記事や意見

---

## 進捗

※タスクごとの完了時にチェックボックスをxにする

- [x] タスク1
- [x] タスク2
- [ ] タスク3

### メモ・課題

- 調査結果や気づいた点
```

1ファイル内で説明が重複して文章が嵩むことは可能な限り避ける

### 3. 完了（done）

`2_wip/` → `3_done/` に移動し、「## 進捗」セクションを削除

```bash
git mv VIBES/2_wip/ファイル名.md VIBES/3_done/ファイル名.md
```

- 「## 進捗」セクション以下を削除
- 完了日を追記（任意）
- 計画のみを残し、後から参照しやすい記録として保管

## 注意事項

- 軽度の作業では計画書を作らず、直接コーディングしても良い
- 計画書の内容は作業中に適宜更新・修正する
- 完了時に計画と実装が乖離した場合は、実際の実装内容を反映する
- doneファイルは、古い情報であるため参照するべきではない(should)
